1. 오늘 배운 내용
   
객체지향 설계 원칙: 강한 결합의 한계와 느슨한 결합의 필요성 학습

의존성 주입(Dependency Injection, DI): 외부에서 객체를 주입하는 방식과 그 장점 이해

제어의 역전(Inversion of Control, IoC): 객체의 생명주기 제어권이 개발자에서 컨테이너(또는 외부)로 이동하는 개념 파악

인터페이스 기반 설계: 추상화를 통해 유연하고 확장성 있는 코드 구조 설계법 실습

생성자 주입: 불변성 확보와 테스트 용이성을 위한 생성자 주입 방식 구현

레이어 아키텍처: Repository - Service - Main(Controller)으로 이어지는 계층 분리 실습

2. 핵심 정리
   
- 강한 결합 vs 느슨한 결합

강한 결합: 클래스 내부에서 다른 객체를 직접 new로 생성하는 것. 하나를 고치면 연쇄적으로 다 고쳐야 하는 '일체형 배터리' 같은 구조라 유지보수가 힘들다.

느슨한 결합: 인터페이스를 통해 소통하는 것. '교체 가능한 건전지'처럼 내부 구현이 바뀌어도 사용하는 쪽은 영향을 받지 않는다.

- 의존성 주입(DI)과 제어의 역전(IoC)

DI: 내가 직접 필요한 물건을 만들지 않고, 밖에서 던져주는 물건을 받아서 쓰는 것.

IoC: 내가 누굴 쓸지 내가 정하는게 아니라, 밖으로 제어권을 넘기는 것. 코드의 유연성이 올라간다.

- 레이어 분리의 이유

Repository: 데이터 저장소 (DB나 파일)에만 집중.

Service: 실제 비즈니스 로직(중복 체크 등)에만 집중.

Main: 전체적인 흐름 제어와 의존성 조립에만 집중.

이렇게 나눠야 각자 자기 일만 잘하면 되고, 나중에 데이터 저장 방식(파일 -> DB)이 바뀌어도 로직(Service)은 손댈 필요가 없어진다.

4. 느낀 점

처음에는 new 하나 안 쓰고 밖에서 넣어주는 게 왜 이렇게 복잡한지 이해가 안 갔다. 

하지만 Mock 저장소로 갈아 끼울 때 Service 코드를 단 한 줄도 수정하지 않고 동작이 바뀌는 걸 직접 확인하니 왜 인터페이스랑 DI를 사용하는지 알게 되었다.

변화에 유연한 설계가 무엇인지 고민해볼 수 있는 시간이었다. 

구조적인 레이어 분리를 어떻게 해야할지 많이 고민할 수 있었다.
